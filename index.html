设计模式系列（二）————单例模式

单例模式应该算是所有设计模式里类图最简单、实现起来代码最短的模式，所以也是各大互联网公司面试的热点。比如，实现一个线程安全的单例模式，单例模式和全局变量的区别。其实，写出一个漂亮的单例模式并非很容易，希望看完这篇文章的人都能立马自己去实现一个漂亮的单例模式。

设计模式系列（一）—— 观察者模式

只掌握继承、接口、多态这些基础的面向对象知识已经不足以设计出优雅、健壮的代码。其实设计模式，说白了就是面向对象的工程师前辈们总结出来的代码组织方式，所有的设计模式都主要为了达到以下三条目的或在三者之间更好地权衡：

  1. 可复用性（相同结构的代码最大化地共用）
  2. 可扩展性（便于增加新需求）
  3. 易维护性（修改某一个模块尽量不牵动或者少牵动其他模块，也成为松耦合）

Java IO设计原理——装饰者模式

第一次看Java IO相关讲解的时候，几乎每个人都会被那么多各种各样的输入输出流的类搞蒙，做一个IO传输需要创建很多个类的实例，让人难以接受。其实JAVA先驱者应用的就是装饰者模式这种设计模式。此文就是为大家解答JAVA IO这么设计背后的原理，并非讲解IO类的使用方法。

装饰者模式的功能和继承类似，都是为了扩展类的功能。继承通过子类继承超类的功能，并添加或覆盖自己特有的功能，这也符合面向对象的思想。但是，简单粗暴地继承会带来最直观的问题就是类爆炸，产生太多太多的类。而装饰者模式，是将需要扩展的这些功能封装成多个装饰类，这些每一个功能在代码运行时候可以灵活地组合，而不是通过继承超类在编译时候决定类的功能。
