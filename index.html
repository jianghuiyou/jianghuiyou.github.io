探秘计算机系统中的文件描述符

发表于 2018-02-22 | 分类于 技术 |

一切皆为文件是Unix系统中重要理念之一，无论是打开的磁盘文件、网络socket还是管道、共享内存、消息队列等这些IPC（进程间通信）通通被抽象为文件，这是通过虚拟文件系统VFS来实现，而我们编写用户程序的时候看到的通常只是一个整形数字代表了文件描述符，那么计算机系统内部是如何表示一个被打开的文件的呢？
我们先理解一下三个重要的概念：

描述符表

每个进程独立占用的资源，也就是用户程序员看到文件描述符的集合，例如标准输入0、标准输出1、标准错误2，每个描述符表项指向打开文件表中的一项。

打开文件表

所有进程共享的当前计算机系统打开的所有文件表项的集合，每一个打开文件表的表项包括当前文件位置、引用计数、以及一个指向inode表项的指针，其代表一个文件被打开后的操作状态集合，多次调用open函数会产生多核文件表项（文件访问的位置、访问权限等等）。而在调用close(fd)关闭描述符为fd的文件时，仅会将其指向的文件表表项的引用计数减一，直到为0才将该表项删除。

inode表

如果说打开文件表项代表一个文件被打开后操作状态，那么inode表项（linux系统中inode在unix系统中被作为vnode）就是这个真实文件元数据状态的集合，包括文件大小、文件各种访问时间、文件类型等多种元数据信息，通过调用stat函数通常可以查看。所以多次调用open函数打开同一个文件时候，就会是多个打开文件表项指向同一个inode表项，如下图所示（参考自CSAPP）。

如何最大程度避免Java空指针异常的坑

发表于 2018-01-22 | 分类于 技术 |

NullPointerException应该是所有Java程序员都遇到过的最简单却也是最容易忽视的异常，由于自己也被它坑害过，所以查阅不少资料总结下避免Java空指针异常的最佳实践。

1. 对所有返回自存在null隐患方法的对象引用进行非空检测

不可信的方法既指来自非null安全的第三方库，也包括自己或者同事写过其他方法，即使当前版本该方法返回的对象引用不可能是null，你也难以确保未来不会有心大之人修改成存在返回null的情况。null安全类典型的就是apache commons库的StringUtils，它的很多方法比如isEmpty、isNumeric等都是null安全的，在字符串String引用处理之前可以多多使用。对象引用用前记得检测null，损失这一点点的性能开销远比代码存在安全隐患更划算。当然你也可以使用断言类型的方式来检测空指针的问题，这属于个人偏好。

2. 使用常量字符串来调用equals()或equalsIgnoreCase()

equal方法本身的自反性让我们常常忽略a.equal(b)和b.equal(a)之间微妙的区别，但在如下情况我们往往会恍然大悟。

在判断一个字符串是否与已知常量字符串相等时候，将其放入equal方法的参数往往可以避免不必要的空指针异常。

3. 尽量避免在函数中返回null

除非不得已，尽量返回一个空的对象也不要返回null，这于人于己都是有益的。对于返回类型是集合的函数，在没有有效结果返回的时候也返回一个size为0的集合或者Collections.EMPTY_LIST等专门用来表示空集合的类。

4. 自动装箱需谨慎

在业务参数众多的情况如果每个均进行非空的检验确实会让代码看起来不美观，有很多第三方库都实现了自己的非空注解来确保该对象引用为非空，感兴趣可以自行google之。在eclipse或idea中也可以使用一些静态代码检测插件，比如FindBugs来检测Java代码中是否存在空指针异常安全隐患。

读《数学之美》有感

发表于 2017-11-28 | 分类于 阅读 |

最近一段时间也读完了几本书，刚刚读完的这本《数学之美》给我带来的感受和触动最深，于是在此记录下自己所想。对于该书的作者吴军博士的了解来自于之前是自己在北邮学活中心听过他做的关于Alpha Go相关分享，并听说过他上一本著名的书《浪潮之巅》，在抱着好奇之心阅读完《数学之美》之后，对于吴军博士的崇敬之情油然而生，能将很多晦涩难懂的计算机、通信领域数学模型用尽可能浅显易懂展现给读者实在难能可贵，贝叶斯公式、维特比算法、马尔科夫模型等等这些本科学习时候感觉枯燥的数学公式在他的讲解下变得更有实际意义，能让我知道这些其实并不复杂的数学模型可以解决很多不可思议的工程问题，大为惊叹的同时深表相见恨晚。如果能在大二大三读这本书，也许能让我对这些数学模型产生更加浓厚的兴趣。

这本书从人类的文字语言谈起，到自然语言处理从基于规则发展到基于统计的历史以及其中的关键数学知识，到建立搜索引擎中网络爬虫、建立索引、PageRank网页质量排名、TF-IDF相关度排名等数学模型和技术，以及分类、加密、输入法中等运用很多技术模型，最后是近几年火热的大数据、人工神经网络等等。虽然本书由于读者受众的缘由涉及到技术和数学的部分并不复杂，但最大的意义在于告诉大家有时简单到谁都理解的数学模型却能解决非常复杂并有实际价值的工程问题，这着实令人激动。其实这也是我认为学习通信、计算机等这些工科的意义所在，就是运用所学为社会创造价值，并不是多发几篇成果仅仅能跑在实验环境的论文。

简单很多时候比复杂更有效，人工神经网络、MapReduce系统等等这些背后的原理其实都又美又简单，就像是苹果的产品，既好用又漂亮。其中谈到托勒密的地心说用了甚至40个圆来拟合行星运行轨道，刚刚出现哥白尼的日心说即一个椭圆的模型更加简洁，但是效果并没有复杂的地心说要精确，但是经过开普勒的数学分析之后才将日心说发扬光大。所以一开始简单正确的道路并并一定比之前错误的道路得到更好地结果，也许只是万事俱备只欠东风，一旦东风到来变会一飞冲天。以后不管是设计计算机系统、还是产品思路，一定先从最简单的思路、最能让用户接受、体验最好的方式考虑。

数学其实可以并不那么复杂，自然的背后蕴藏的真理就是简而美的数学。

并发世界探索（二）—— 读写锁

发表于 2017-11-06 | 分类于 技术 |

读写锁本质上是任意时刻只允许一个写者或者多个读者操作的自旋锁，C++17之后出现std::shared_mutex就是读写锁的实现，不过为了加强对并发技术的理解，自己基于C++11中的互斥量和条件变量来实现一个读写锁，并且通过简单修改就可以实现读者或者写者优先以及随机优先级等变形。

条件变量主要应用在当某条件满足时候，就进行wait或者notify。比如在此读写锁中，对于写者上锁操作，当有其他读者或写者存在时候，则需要该写者进行wait等待；而当写者释放锁的时候就会通知其他读写者可能满足它们的持锁条件。而条件变量常常又结合互斥锁来使用，互斥锁产生的临界区使得对变量操作可以是线程安全的。废话不多说，先上代码。

整数变量readers和布尔变量writer分别代表当前正在执行的读写者，而reader_waiter和writer_waiter则是记录着当前等待的读写者个数。这里将它们设置为volatile是为了防止编译器或硬件系统将这些变量优化到cache或寄存器中，导致多核系统中操作这些变量时候的不一致性，这样所有的操作都是从物理内存中进行存取。

上面读写锁是一个写者优先的情况，对write_unlock中读写者条件变量的顺序颠倒就可以变成读者优先，或者添加随机数来实现较为公平的随机化优先级形式。
