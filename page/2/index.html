HTTP长连接如何判断数据传输结束

发表于 2017-04-23 | 分类于 技术 |

最近在流量分析项目中，针对http数据由于keepalive的存在，如果根据TCP状态机进入CLOSE才判断数据结束，会存在着较大的延迟，在linux系统中抓包实测大概会有60秒左右TCP的持续延迟，这就大大妨碍系统分析数据的时效性。为了解决这个问题，自己对HTTP长连接进行研究并实现了数据提前抽取的实时性功能。

KeepAlive属于HTTP头部Connection字段，即Connection: KeepAlive。它是在HTTP1.0版本中引入，并在HTTP1.1中默认配置的属性，现在大多主流的浏览器都已经是HTTP1.1，引入KeepAlive的主要目的就是降低web应用中http传输延迟和带宽。

传统的HTTP0.9版本（已废弃）和没有进行keepalive设置的1.0版本，当进行一次HTTP的请求响应的数据等信息的传输都需要建立一个TCP Connection，也就是要走三次握手、四次挥手并严格遵从TCP拥塞控制机制，一旦数据输出完TCP会立即关闭，而如果再次进行数据传输还需要再次建立TCP Connection，这样引入KeepAlive机制的优势就不言而喻。KeepAlive确实很棒，但在项目实时性需求下又成了麻烦之处，还好HTTP丰富的头部信息为我们解决这个问题带来了帮助。

HTTP数据传输主要分成两种模式，一种是通过Content-Length，另外一种是通过Transfer-Encoding: chunked，这两种各有千秋。

这篇文章主要介绍基于比较的经典排序算法中的插入排序和快速排序，基于比较的排序需要用 \(a_i\le a_j\)或者\(a_i\ge a_j\)类似的逻辑比较来决定两个数的大小。这类排序算法的渐进下确界是 \(\Omega(nlgn)\)，可以使用决策树模型来进行证明，决策树是一个完全二叉树，\(n\)个数共有\(n!\)个排列方式，对应的叶子结点也至少要有\(n!\)个，而最坏的比较次数对应这个决策树的高度 \(l\) ，由此可得 \(n!\le l
\le 2^h\)。两边取对数之后可得\(h \ge log(n!) = \Omega(nlgn)\)。更加详细的证明过程以及各种渐进界的介绍请参考算法导论。

对nodejs异步IO、事件轮询的理解

发表于 2016-11-22 | 分类于 技术 |

nodejs是JavaScript这门编程语言在服务器端运行的平台，将chrome浏览器中的js引擎V8移植到服务端后，这使js俨然成为一种全栈语言。在实践中自己已经开始编写node异步读写数据库IO的操作，并应用了promise特性。而异步IO、事件轮询作为nodejs最突出的特点，也可以算是最大亮点，不得不好好理解一下。

IO操作通常指网络IO、磁盘文件读写、各种数据库读写以及定时器操作等等，而这种操作相对于CPU的运算速度相当耗时。而异步IO就是在发生上述IO操作的时候，原来的程序不等待返回结果直接继续运行，完全不鸟这个IO操作的过程。这种机制的好处就是大大提升CPU的利用率，让CPU可以尽可能马不停蹄地工作，不需要等待慢腾腾的IO操作，尤其适合IO密集型应用。这也是一个很简单的node服务器可以支撑上千并发访问量的原因。
